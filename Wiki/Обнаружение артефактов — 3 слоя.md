# Обнаружение артефактов — 3 слоя

> **Суть:** политика ревью (персоны/праймеры/вейверы) **версионируется рядом с кодом**
> (repo-aware, первопринцип из [[MOC — ai-reviewer]]). Обнаружение трёхслойное с
> приоритетом, дедуп по ID.

## Три источника с precedence (`scanner.go:39`)
1. **Закоммиченный Markdown на head-SHA** — источник истины, едет с репозиторием.
   `git ls-tree` + `git show <SHA>:<file>`; учитываются выделенные директории
   `.ai-review/<repo>/<type>s/` или файлы с frontmatter `ai_review:`.
2. **Локальные repo-scoped директории** `.ai-review/<owner>/<repo>/{personas,primers,waivers}`
   — машинные / CI-оверрайды.
3. **Свободные файлы в search paths** (требуют тег `ai_review:`) — ad-hoc определения.

**Дедуп по ID:** первый источник побеждает (`scanner.go:66`). Семантика:
`committed > local > loose`.

## Идея — двухшаговый парсинг frontmatter (`scanner.go:188`)
Сначала **дешёвый байтовый скан** на `ai_review:`, и только при попадании — дорогой
YAML-парсинг. ~99% `.md`-файлов отсеиваются до парсинга. ID при отсутствии = имя файла
без расширения.

## Почему repo-aware (из VISION.md)
> «Different repositories have different concerns. A blockchain repository, a web app, an
> SDK... should not all be reviewed as if the same checklist applies.»

Праймеры объясняют архитектуру, персоны фокусируются на проектных рисках, вейверы кодируют
политику исключений — и всё это **эволюционирует вместе с кодом**, а не как
ambient-поведение машины.
