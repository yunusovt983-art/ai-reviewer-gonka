# Primer и Concept — инъекция знания

> **Суть:** праймер — кусок доменного контекста, который подмешивается в промпт
> [[Persona — корень агрегата ревью|персоны]] **только когда его фильтр совпал** с
> ревьюируемыми файлами. То, чего нет в диффе, но без чего ревью слепо.

## Primer (`primer.go:3`)
```go
type Primer struct {
    ID, Type          string
    Filters           FilterSet  // тот же предикат, см. [[FilterSet — управление стоимостью]]
    AuthoringConcepts []string   // семантические теги
    Content           string     // Markdown, инъектируемый в промпт
}
```

### Живой пример (`gonka/primers/inference-chain.md`)
Объясняет модели, что Gonka — live-блокчейн на Cosmos SDK, что вес/награды/право голоса
определяются «Proof of Compute», как регистрируются инференсы. Это **доменный blueprint**,
без которого LLM не поймёт инвариантов.

## Concept — семантический слой поверх путей
```go
type Concept struct {                 // context_concepts.go:19
    Name    string
    Sources []ConceptSource           // обратные ссылки на праймеры
}
```
Идея: связывать праймеры не только по **путям файлов**, но и по **смыслу** изменения.

### Правило матчинга (`context_primers.go:86`)
Праймер срабатывает, только если выполнено **И**:
- **фильтр совпал** (или пуст), **И**
- **концепты пересеклись** (если объявлены у праймера и у запроса).

Это предотвращает ложные срабатывания «только по концепту» или «только по пути».

## Наружу: подкоманды `context` / `concepts`
Отдельные CLI-команды отдают сопоставленные праймеры/концепты для внешних интеграций
(`PlannedContext{Files, Functions, Concepts}`) — можно спросить «какой контекст
релевантен этому плану изменений» без полного прогона.
